Detectable Anomaly Types

Amount Outliers

  • Payable above historical mean for category/supplier
  • Unusually high fees/interest (multa, juros)
  • Suspicious discounts (desconto >> typical)
  • Detection: 3σ from rolling 6-month mean per category

Timing Outliers

  • Payments much earlier/later than typical
  • Payables on unusual dates (weekends, holidays)
  • Receivable overdue far longer than typical
  • Compare data_vencimento vs historical pattern

Behavioral Outliers

  • New supplier with unusually large first payment
  • Customer method change (BOLETO → PIX)
  • Multiple partial payments on single installment
  • Renegotiated installments with significant value changes

Category / Cost Center

  • Category spending spike vs previous months
  • New category appearing unexpectedly
  • Cost center allocation pattern changes
  • Unreconciled items for extended periods

Honest Assessment

Every part is buildable, but none comes pre-built. The API is a data access layer, not an analytics engine. No SUM, GROUP BY, or MoM comparison. But filters give precise control.

Recommendation

Build a thin aggregation microservice that runs 4–5 queries daily, caches results, exposes a /summary endpoint. Feed into LLM/template for natural-language output. ~5 API calls per summary.

API Gap — No Write Access for Notifications

The 3 Financial API POSTs create cost centers, receivables, and payables only. None trigger notifications.

  • Workaround: Build your own sender using API data + external channel (SendGrid, Twilio).
  • Caveat: Self-sent notifications won't appear in Conta Azul's history.

Rate Limits — Undocumented

HTTP 429 returned but threshold not documented. Start at 2–3 req/sec, exponential backoff on 429. Use incremental sync via data_alteracao_de/ate. Cache static data locally.

Efficient Patterns

  • Set quantidade_registros=1000 on financial endpoints
  • Use multi-ID filters to combine N queries into 1
  • Incremental sync via data_alteracao_de/ate
  • Cache categories, cost centers, accounts

Known Bottlenecks

  • Sales API: 50/page max — 1,000 sales = 20 pages
  • parcelas/{id}: 1 per call — N+1 problem
  • No parallel batch creates for financials
  • Contracts: 50/page, limited filtering
Cross-Cutting: API Limitations

Key Gaps Affecting All Use Cases

  • No webhooks — must poll for new/changed data. Real-time triggers require periodic sync.
  • No server-side aggregation — all sums, averages, and distributions computed client-side.
  • No audit log — no "who changed what" data. Limits anomaly root-cause analysis.
  • No budget data — cannot compare actual vs planned. Limits anomaly context.
  • Rate limits undocumented — HTTP 429 returned at unknown threshold. Build with exponential backoff.
  • No feedback API — ML models can't learn from user corrections via the API. Build feedback loop externally.
  • No notification write access — read-only for all reminder/collection data. External sending required.

Mitigation Strategy

  • Use data_alteracao_de/ate for incremental sync instead of full pulls
  • Cache categories, cost centers, and pessoa data locally (changes rarely)
  • Set quantidade_registros=1000 on financial endpoints to minimize pages
  • Build your own feedback/correction layer outside the Conta Azul API

4–6 weeks · Needs historical depth
Anomaly narratives layer LLM on Phase 2's outlier detection. Churn and seasonal patterns require 6–12+ months of accumulated data. Start the pipeline early, activate models when data matures.

Payment Reminders + Batch Optimization
Ongoing · External integration
