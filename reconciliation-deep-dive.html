Bank Reconciliation — Deep Dive

Complete analysis of reconciliation capabilities across Conta Azul API specifications

~60%

Native Integration + Read-Only API

Conta Azul handles bank statement import and transaction matching natively via automatic banking integration with 13+ Brazilian banks. The API exposes rich reconciliation data (flags, IDs, origins, balances) but provides no write endpoints for reconciliation operations. The real gap is narrower than it first appears: you can't drive matching via API, but you don't need to — Conta Azul already does it automatically.

Reconciliation Flag on Parcela

conciliado: boolean on every installment record. Tells you if it's been reconciled in the ERP. Not writable via PATCH — the PATCH endpoint only accepts: nota, descricao, vencimento, composicao_valor, versao, metodo_pagamento, nsu, pagamento_agendado, id_conta_financeira.

Reconciliation ID on Baixa

id_reconciliacao: UUID on every payment record. Links to the internal ERP reconciliation record. Read-only — no endpoint to create or update reconciliation records.

Payment Origin Tracking

The origem enum on Baixa records tells you HOW a payment was created — critical for knowing which came from bank imports vs manual entries. Key reconciliation values: SALDO_CONTA_BANCARIA, IMPORTACAO_DOCUMENTO.

Sales Unreconciled Count

quantidade_nao_conciliados in Sales Totals (Totais) tells you how many sale items still need item-level reconciliation. The CONCILIACAO_ITENS pending status flags sales waiting for reconciliation.

NSU / Bank Transaction Reference

nsu: string on Baixa records stores the bank transaction reference number. Writable via PATCH parcela, but the reconciliation link itself cannot be established through the API.

Conta Azul's Native Banking Integration

Bank statement import and reconciliation are handled natively inside the ERP — no API or browser automation needed for these steps.

Automatic Bank Integration (13+ Banks)

Daily automatic import of bank transactions. Supported banks: Banco do Brasil, Bradesco, BS2, BTG Pactual, Caixa Econômica, C6 Bank, Cora, Inter, Itaú, Nubank, Santander, Sicoob, Sicredi. PagSeguro & Stone integrate card machine transactions only. No limit on number of connected accounts.

Manual OFX Import (Any Bank)

For banks without automatic integration: upload OFX files (Money 2000 format, max 5 MB, ascending date order, D-1 or earlier). Navigate to Financeiro > Contas financeiras > Ações da Conta > Importar extrato (.OFX).

Automatic Reconciliation Matching

Configurable matching rules: Exact (same amount + same due date) or Approximate (same amount + due date within ±5 days). Exact runs first when both are enabled. Duplicates (identical amount & date) are skipped and left for manual review.

Auto-Payment Processing

Recognized payments automatically update payables (baixa do contas a pagar) and receivables. Cross-references incoming payments with invoices. Saves up to 20 hours/month per Conta Azul's estimates.

Remaining API Gaps

With native integration handling import and matching, the actual gaps are narrower:

Bank Statement Listing

No API endpoint to query imported bank transactions. You can only see the resulting payments (baixas) after matching — not the raw bank statement entries. For monitoring purposes, this is usually sufficient.

Manual Match Override

No API for proposing, confirming, or rejecting matches. When auto-matching fails (duplicates, partial amounts), the ERP user must resolve in the UI. Browser automation needed only for these edge cases.

Set Reconciled Status

Cannot mark parcelas as conciliado: true or assign id_reconciliacao to baixas via API. These fields are system-managed. The PATCH parcela endpoint explicitly excludes both fields.

Integration Configuration

Cannot connect/disconnect bank accounts or change matching rules via API. Initial setup and configuration changes must be done in the Conta Azul UI. Once configured, everything runs automatically.

Recommended: Let Conta Azul Reconcile, Monitor via API

Leverage native integration, monitor with API

Since Conta Azul handles import and matching natively, the best strategy is to let it do its job and use the API for monitoring and exception handling:

  1. Set up automatic banking integration in Conta Azul UI (one-time, per bank account)
  2. Configure matching rules — enable both exact and approximate (±5 days) for best coverage
  3. Monitor unreconciled items via API: query parcelas with conciliado=false using GET .../contas-a-receber/buscar and GET .../contas-a-pagar/buscar
  4. Track reconciliation progress by polling conciliado flag changes via data_alteracao_de/ate
  5. Verify balances via GET /v1/conta-financeira/{id}/saldo-atual to flag discrepancies
  6. Alert on exceptions — items unreconciled for 7+ days, balance mismatches, duplicate amounts
  7. Store external refs on matched parcelas via PATCH .../parcelas/{id} using the nsu field

When browser automation IS still needed

  1. Initial bank connection setup — one-time per account, requires bank-side authorization
  2. Unresolved matches — when auto-matching fails (duplicates, split payments), a human or browser agent must resolve in the UI
  3. OFX import for unsupported banks — if the bank isn't in the 13+ homologated list, OFX must be uploaded manually via the UI
  4. Changing reconciliation configuration — matching rules can only be adjusted in the ERP settings

Key insight: the gap is much smaller than it appears

With native banking integration covering statement import and auto-matching, the only reconciliation work left for an agent is exception handling — roughly 5–15% of transactions that don't auto-match. For those edge cases, browser automation (Playwright or Claude Computer Use) can navigate the reconciliation UI. The API's read-only conciliado flag tells you exactly which items need attention.
